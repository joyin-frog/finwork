# Spec: 单据拍照 → 金蝶凭证草稿

## 背景与目标

月末大量纸质手写单据（付款单、费用报销单、电子汇票）积压，逐张手录金蝶耗时。

**目标**：用户选择一个装满单据照片的文件夹，agent 批量 OCR + 识别字段 + 推断科目，在对话框输出结构化凭证草稿，用户逐张核对后照着填金蝶。

**不做**：直接写入金蝶（API 权限后期再开）、自动过账、替用户做最终判断。

---

## 单公司假设（架构边界）

**本功能假设一个安装只服务一家公司**（用户确认：一次只管一家）。

- 科目表存 `app_settings.kingdee_chart_of_accounts` 单例，**换公司 = 重新导入覆盖**，这是预期行为，非 bug
- 对照表放知识库、公司名从 profile 读，同为"当前一家公司"假设
- **不做多账簿/多公司隔离**（保持简单，避免为不确定的未来编码）

**换公司操作流程**（写进 skill 提示）：
1. 重新导入新公司科目表（覆盖旧的）
2. 替换/删除旧对照表文件（否则旧科目编码串味）
3. profile 改公司名

安全兜底：即使旧对照表未清，`query_kingdee_accounts` 会验证编码——旧编码在新科目表不存在则标 ⚠️，不会错记。

## 科目表结构（来自真实导出）

金蝶云星空导出的科目表 Excel 表头 10 列：

`主表项目(减少) | 主表项目(增加) | 编码 | 名称 | 全名 | 余额方向 | 科目类别 | 外币核算 | 核算维度 | 数据状态`

- 283 行，费用/损益类 118 个明细，最深三级（如 `6602.24`）
- **核算维度是科目自带属性**：6602.xx→部门，1002.xx→银行账号，应收/预付→客户/供应商，个人往来→员工
- 费用科目极细：6602 管理费用下 29 个明细、6601 销售费用 20 个；差旅费在 5301.04(研发)/6601.04(销售)/6602.04(管理) 各有一份 → **落哪个科目靠维度=部门区分**

**存储影响**：`KingdeeAccount` 需加 `dimension` 字段（维度类型：部门/供应商/客户/员工/银行账号/无），import 工具解析"核算维度"列。确定科目编码后维度类型即确定，只剩维度的**值**（哪个部门）要从单据/对照表取。

| 单据类型 | 识别特征 | 借方科目 | 贷方科目 |
|---------|---------|---------|---------|
| 费用报销单 | 表头含"报销" | 管理费用-XXX / 销售费用-XXX | 银行存款 / 库存现金 |

## OCR 技术选型：三层结构

实测结论（2026-07-01，2 张真实手写报销单）：rapidocr 对**印刷体**（单据类型、日期、报销项目编号）识别准，对**手写摘要/被红章压住的金额**基本不可用。据此定三层：

```
第1层（默认）：rapidocr —— 本地、离线、CPU 秒级、14MB
  读印刷体线索：单据类型、日期、编号、科目关键词
        ↓
第2层（核心）：金额勾稽校验 —— 不依赖 OCR 读准金额
  用单据自带的冗余（明细Σ=合计=大写）交叉验证
        ↓
第3层（按需）：Claude vision —— 仅当勾稽不平/手写糊到救不回时点用
  手写识别明显强于任何本地 OCR
  数据通道 = 现有 claude-adapter 附件通道，不额外破坏隐私
```

### 已排除方案（勿重复评估）

- **重型 VLM OCR（Unlimited-OCR / DeepSeek-OCR / PaddleOCR-VL 等）**：需 NVIDIA GPU + CUDA，模型 GB 级，vLLM/SGLang 服务器部署。与"Tauri 桌面端内嵌 CPU Python、离线、轻量分发"架构完全冲突。手写识别虽强，代价不可接受。
- **vision 作为默认层**：会让每张单据都走云 API（成本 + 隐私面扩大），故降为"按需兜底"而非默认。

## 金额勾稽校验（金额处理的核心机制）

**原则：AI 不负责"读准"金额，负责"校验"金额。** 金额是零容错字段，读错比不读更危险；但单据自带冗余，用数学矛盾暴露问题，最后由用户拍板。

一张单据里金额通常出现多次且必须自洽：

```
明细逐行相加  =  合计数字  =  大写金额
  1377+2323   =   3700    =  叁仟柒佰
```

校验逻辑：

1. **抽取所有金额线索**：每行明细数字、合计数字、大写金额
2. **勾稽比对**：
   - 三者一致（差额 < 0.01）→ ✅ 高置信度，自动通过，不打扰用户
   - 任一不一致 → ⚠️ 标记，**精确指出哪一处对不上**，展示全部候选值，用户拍板
3. **无效字符容错**：手写常有涂改/废笔（如"~~拾X万~~叁仟柒佰"）。不要求 AI 判断哪个笔画有效——**若把废笔读进去，勾稽自然不平 → 触发人工 → 用户判定废笔**。让数学矛盾兜底，不靠 AI 猜。
4. **只有一处金额线索**（无明细/无大写可比）→ 无法勾稽 → 一律 ⚠️ 人工确认
5. **金额为 0 或负数** → 报错，不生成凭证行

---

## OCR 预处理规则

1. **图片旋转**：调用 rapidocr 前检测方向自动旋转（`use_angle_cls=True`）
2. **支持格式**：`.jpg` `.jpeg` `.png` `.webp`
3. **文件过滤**：跳过非图片文件（隐藏文件、`.DS_Store` 等）
4. **单张超时**：OCR 单张 > 30s → 跳过并标记失败，继续处理其余
5. **图片质量提醒**：清晰度直接决定勾稽能否读全；批处理前提示用户尽量拍清楚、金额栏不被反光/褶皱遮挡

## 科目映射推断顺序

```
1. search_knowledge("科目对照表 [收款方或费用类型]")
   → 有精确匹配 → 直接用，置信度=高
   → 有模糊匹配 → 展示建议，置信度=中，标黄

2. 无知识库结果 → LLM 根据描述 + 科目表推断
   → 置信度=低，标橙，要求用户确认

3. 推断失败 → 科目留空，标红 ⚠️
```

映射时必须调 `query_kingdee_accounts` 验证科目编码存在，不存在的编码不得输出。

### 对照表：从零开始，自动沉淀

- **初始为空**：不预填种子，第一版全靠 OCR + LLM 推断 + 用户确认
- **自动追加**：用户确认科目后，agent 自动往对照表文件追加一行，并**明示写了什么**——
  `已记入对照表：杰强 → 6602.24 劳务费 → 综合部`
  用户一眼可见，记错当场纠正
- **越用越准**：下次遇到同一关键词，对照表命中 → 高置信度直接建议
- 追加动作对用户透明，不静默写入

## 交互模式：汇总确认（用户 2026-07-01 定案）

**不逐张打断,先全部识别完,出一张汇总大表,用户从中挑有问题的行集中比对确认。** 理由:注意力一次集中,能快速扫出⚠️行;逐张弹窗打断节奏。

### 汇总大表

所有单据识别完,一次性展示:

```
共 8 张单据 · ✅ 5 张自动通过 · ⚠️ 3 张待确认

| # | 单据    | 日期     | 摘要        | 科目编码 | 科目名称        | 维度   | 借方     | 贷方     | 状态 |
|---|---------|----------|-------------|----------|-----------------|--------|----------|----------|------|
| 1 | 报销单  | 2026/6/1 | 住宿+餐饮   | 6602.04  | 管理费用-旅差费 | 综合部 | 3,700.00 |          | ✅   |
| 2 | 付款单  | 2026/6/1 | 付杰强劳务  | 6602.24  | 管理费用-劳务费 | ⚠️     | 137,700  |          | ⚠️维度 |
| 3 | 报销单  | 2026/6/1 | 物冷购      | ⚠️       | ⚠️              | ⚠️     | ⚠️金额   |          | ⚠️多项 |
| … |
```

置信度标记:✅ 高(勾稽通过/对照表命中,自动用) · 🟡 中(建议确认) · 🟠 低(必须确认) · ⚠️ 缺失或冲突(必须人工)

### 集中确认

用户挑出⚠️行后,把这些行的待确认项**聚合成一个多问答浮层**(见下节),左右切换逐项确认,一次提交。不是每个⚠️弹一次。

确认完 → 回填大表 → **生成 Excel**(对照手填清单 + 存档,非金蝶导入模板;金蝶无导入入口)。

## 完整流程

```
用户：选择文件夹（Tauri open({ directory: true })）
  ↓
run_python → 列出目录下所有图片（按文件名排序）
  ↓
逐张识别（顺序执行，非并发）：
  ├─ ocr-image → 原始文字（rapidocr；糊到不可用时按需 vision）
  ├─ LLM 提取字段（单据类型/日期/金额线索/收款方/摘要）
  ├─ 金额勾稽校验（明细Σ=合计=大写；不平标⚠️）
  ├─ 科目映射（对照表命中 → LLM 推断 → 空）
  ├─ query_kingdee_accounts 验证科目 + 带出维度类型
  └─ 生成该行草稿（自动通过 or 标⚠️）
  ↓
全部识别完 → 汇总大表（✅N 张 / ⚠️M 张）
  ↓
用户挑⚠️行 → 多问答浮层集中确认（左右切换）→ 一次提交
  ↓
回填大表 → 自动追加对照表（明示写了什么）
  ↓
生成 Excel（对照手填清单 + 存档）→ 用户照表填金蝶
```

## 错误处理

| 情况 | 处理方式 |
|------|---------|
| OCR 返回空 | 标记"图片无法识别"，跳过，继续其余 |
| 单据类型无法判断 | 科目全部留空，标 ⚠️，用户手填 |
| 借贷不平衡 | 输出草稿但顶部加红色警告，不阻止展示 |
| 科目编码在科目表不存在 | 清除编码，标 ⚠️，给出最近匹配建议 |
| 图片方向识别失败 | 原样送 OCR，结果可能不准，标注"可能识别偏差" |

## 待扩展:多分录凭证 + 银行回单(明天材料后)

**重要修正**:当前 `export_kingdee_draft` 分录是「一借一贷成对」,但真实凭证是**多行独立借贷**——一张单据可有多个费用明细→多个借方行(不同科目/不同维度值),对应一个/多个贷方行(银行/现金),Σ借=Σ贷。这正是金蝶凭证录入界面的表格结构(序号|摘要|科目编码|科目全名|核算维度|借方|贷方)。

真实数据结构:
```
一张单据(报销单/付款单)
  ├── 背后夹 K 张发票/附件      → 附件数
  ├── N 个费用明细(住宿+餐饮…)  → N 个借方行,各自科目+维度值
  └── 银行回单(CSV/Excel,多条) → 付款依据,对应贷方
        ↓ 一单一凭证
凭证 entries: [{summary, account, accountName, dimensionType, dimensionValue, debit?, credit?}...]
约束: Σdebit == Σcredit
```

要改/新增:
- `export_kingdee_draft` 分录结构:成对 → **多行独立借贷**(每行要么借要么贷)
- `map_voucher_account`:逐费用明细各自映射(工具够用,编排层多调几次)
- **银行回单匹配**:复用 `reconcile_bank_statement`(金额+日期+方向勾对);前置 run_python 把回单 CSV/Excel 整理成 date/amount/direction
- 附件数:一张单据背后 K 张发票 → attachmentCount

已就绪不用改:`check_voucher_amount` 勾稽已支持多明细(明细Σ=合计);`KingdeeAccount.dimension` 维度类型已实现。

已确认规则(用户 2026-07-01):

- **多明细**:以单据分行写明的费用明细为记账单元,每行一个科目;背后发票是佐证非记账单元。维度值默认整单一致(报销部门),逐行可覆盖。
- **餐饮费**:默认→差旅费(差旅频率远大于招待,且差旅伙食全额扣除合规);仅明确宴请客户→业务招待费(6602.06)。
- **合并付款**:频率低,默认一对一,不做多对一优化;出现合并(回单额>单据额)标⚠️人工。
- **预借款冲销(关键,靠单据「原借款」栏判断)**:
  ```
  「原借款」栏 空/未填 → 普通凭证:借 费用(多行) / 贷 银行存款(合计)
  「原借款」栏 有金额   → 冲销凭证:
      借 费用(多行,按明细科目+部门维度)
      贷 其他应收款-个人往来(1221.03,维度=员工=报销人)  金额=原借款
      差额: 应退款→借 银行/现金; 应补款→贷 银行
  ```
  OCR 必须提取「原借款」和「应退(补)款」两栏;有钱员工不借款走普通分录,借款的走冲销。
- 冲销科目 1221.03 其他应收款-个人往来,维度类型=员工(科目表已有)。

## 验收标准

- AC1：选文件夹 → 5 张图片全部 OCR，无一张因上传大小报错
- AC2：横拍图片自动旋转后识别率与正拍相当
- AC3：金额勾稽——明细Σ=合计=大写三者一致时自动通过；任一不一致时标 ⚠️ 并精确指出对不上的那一处、展示全部候选值
- AC4：科目编码输出前必经 `query_kingdee_accounts` 验证，不存在的编码不出现在草稿
- AC5：借贷不平衡的凭证草稿必须有红色警告
- AC6：OCR 失败的单据跳过后继续处理其余，不中断整批
- AC7：仅一处金额线索（无从勾稽）的单据一律标 ⚠️ 人工确认，不自动通过
- AC8：含涂改废笔的金额（勾稽不平）触发 ⚠️，不由 AI 擅自判定哪笔有效
- AC9：多问答浮层——一次下发 N 题，左右切换独立作答，一次提交回填全部；单题路径不回归

4. **汇总/错误处理**
