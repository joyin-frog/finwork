# 财务助手 Agent · 功能需求纪要(用户口述整理)

> 来源:与公司财务人员的一段对话。本文档已剪除口语重复、自我纠正与"为什么这么干"的背景叙述,只保留对 Agent 实现有用的需求、优先级与流程约束。优先级判断与交互约束尽量忠实原话。

---

## 一、总体愿景:八大功能

| # | 功能 | 当前定位 |
|---|------|----------|
| 1 | **通用功能**:文档归纳(合同 / 订单提炼 + 备查) | **P1 先做** |
| 2 | 财务做账记账 | 暂缓 |
| 3 | 资金管理 | 暂缓 |
| 4 | **税务筹划**(含研发费用加计扣除核查) | **P3 独立重要模块,正在细化** |
| 5 | 财务合规与内控 | 暂缓 |
| 6 | 预算 | 暂缓 |
| 7 | **财务分析 / 经营分析** | **P2 先做** |
| 8 | 成本管理 / 成本核算 | 明确暂不做(见下) |

---

## 二、当前节奏:只做三件,做稳做厚

- **立即做**:P1 合同/订单归纳、P2 经营分析 —— "先把这两个做稳、做大、做丰富一点"。
- **紧随其后、独立成模块**:P3 税务筹划(因重要,单独拆出,本纪要已细化)。
- **暂缓**:发票、薪资计算、税费计算等,排在三大件之后。
- **明确排除——成本核算**:依赖订单 / 采购 / 库管数据且需对接 ERP,目前无数据源、连不上,故不做。
- **核心原则**:不一次吃成胖子,每个功能先做核心、跑稳,再迭代加厚。

---

## 三、P1 通用功能 = 合同 / 订单「提炼 + 备查」

### 目标
原件只为日后备查;真正的价值是把关键要素**提炼成结构化总览表**,打开总览即知"本月要付/收哪些款、哪些单子还缺合同/发票",不必每次翻原件。

### 数据结构
- **原件(合同 / 订单 / 附件)单独存档** → 进知识库,仅供备查。
- **总览 Excel**:一行一份合同 / 订单,沉淀关键要素;知识库也有总览入口。
- 附件与提炼分离:订单原件本身常是 Excel,后面以**附件形式**挂合同/单据;附件单独存,要素提炼进总览。

### 合同总览 · 关键字段
甲方、乙方、付款方式、**付款时间**、**开票时间**。
- 例:金蝶服务合同——签订后约定每年 12/31 付款,提炼后打开总览即知 12 月要付哪些款。
- 适用对象:商务合同、**水电等周期性付款**、**分期付款**。

### 订单总览 · 关键字段
甲方、乙方、签订时间、**订单号**、**合同是否签订**、**是否开发票**、**是否交付**。
- 用途:单子快完成时,看"合同是否签订"决定去补签;看"是否开票/交付"跟进闭环。
- **状态可回写**:如由"未开票"→ 开票后,总览须更新。
- 背景(仅说明痛点,不展开):目前靠人工逐单盯"开票没、签合同没",易混乱;本总览即为把这套人工跟踪结构化、可优化。

### 提醒
- **暂不做提醒**,先做到"总览可见、自己看"。
- 迭代项:后续可对**付款 / 收款**加提醒。

## 四、P2 经营分析

### 输入
- 用户从**金蝶导出财务报表**,再导入 Agent。
- 形态:**一个 Excel,内含多个 sheet**——常态 3~4 个(标准财务报表),常扩展到 7~8 个(用户自制表,如**纳税情况**、**业务情况**,这些标准报表里没有)。

### 分析框架:四大能力
**盈利能力 / 营运能力 / 偿债能力 / 发展能力**(ROE、ROI 等指标)。

### 三重基准对比
1. **同业**(行业基准)
2. **去年同比**
3. **预算**

→ 据此判断趋势是否正常。
例:年预算 1 亿,Q1 仅实现 500 万,但同比正常、同业相近 → 属阶段性差异,而非异常。

### 明细下钻
- **收入**:按事业部拆解(如 4 个事业部,各含若干业务线),看各块构成。
- **成本**:料 / 工 / 费各多少、占比多少。
- **税**:近期多交了多少税。
- **期间费用**:财务费用 / 管理费用 / 销售费用三块;**每块下约 10 个明细,明细再逐项分析**。

### 输出
**同时产出 Excel 表 + Word 文档**。

### 交互流程(关键约束)
- **不一步到位生成**:先以文字逐步回答分析(类似对话式长文输出),用户可追问;确认后再让其生成 Excel / Word。
- **生成过程中有疑问 / 不确定,必须提问,待用户确认后再执行**。
  - 理由:若直接生成且结果不对,用户不会逐项核查,错了也看不出来;放在对话里用户能看出哪里有问题。

## 五、P3 税务筹划(独立重要模块)

### 定位
根据公司的**行业 / 业务 / 规模 / 地区**,匹配可享受的**税收优惠、政府补贴、出口/产品退税**等政策,并持续做**政策速达**。

### 流程
1. **先做"企业画像"问卷**(约 10 题),设定前置条件,例如:
   - 小规模纳税人 vs 一般纳税人
   - 是否高新技术企业
   - 所在地区(如上海松江区 / **临港新片区** / **浦东新区**——各有专属补贴)
   - 年营收规模
   - 产品类别(如是否电子产品)
2. **按画像 + 地区检索政策**(国家级 + 地区级),把最新政策带给企业。
3. **政策速达**:国家或地区政策变动 / 新政出台时,**主动检索并推送**,无需用户自己满世界搜。
   - 信息来源:**以国家税务局为主**;可对接税务局 / 相关公众号等信息源。

### 价值锚点
用户现状是"听到的政策才去享受,没听到就当它不存在" → **Agent 的核心价值 = 主动发现用户不知道的政策**。

### 子模块:研发费用加计扣除核查(高新企业;国家审核极严)
- **目的**:核查用户**做得对不对**——是否用了最新政策、**归集范围**是否有问题。
- **输入**:用户上传**立项报告**、数据**归集范围**等资料。
- **动作**:Agent 检索现行政策,对上传资料做**合规比对校验**,按要点提示缺漏。
- **智能提示示例**:研发费用含**委外**部分 → 提示"委外是否已在相关部门**备案**?"(企业常不知此要求,往往审计时才暴露)。
- **应输出**:能享受哪些政策 / 必须保证哪几点 / 哪些范围要归集 / 需要哪些法规支撑文件。
- **本质**:相关法规客观存在但企业不懂 → Agent 用政策对上传资料做合规校验 + 政策速达。

## 六、跨功能产品原则(从对话提炼)

1. **不一次吃成胖子**:每个功能先做核心、跑稳,再迭代加厚。
2. **生成式动作"先问后做"**:有疑问先提问、用户确认后再执行(经营分析、税筹这类不可盲信的输出尤其如此)。
3. **提炼优先于堆原件**:原件只为备查,价值在结构化要素总览。
4. **政策类信息要"主动速达"**,不依赖用户自己搜。
5. 用户判断:**这些都适合做成 skill**。

---

## 七、明确暂缓 / 排除清单

- 提醒功能(先靠总览自看,后续再加付款/收款提醒)
- 成本核算(缺 ERP / 订单 / 采购 / 库管数据源)
- 发票、薪资计算、税费计算等(排在三大件之后)
