# ADR 0002 — 多智能体协调模型：数据层共享 + 主 Agent 编排，禁止 A2A

状态：已接受（2026-07-02）
关联：spec-role-registry.md（角色注册表/调度史）、spec-cockpit-v3.md（智能体空间）、红线 3（数据信任链）/ 红线 5（写操作审批）

## 背景

六个预制角色（spec-role-registry）落地后，需要回答：跨职能域的财务工序（工资→个税申报复核、月末结账的跨域检查、合同→收付提醒）中，角色之间如何协调与共享信息？是否需要工作流引擎或角色间通信？

## 决策

协调发生在**两个层面**，都不是角色间通信：

1. **信息共享走数据层，协议 = 信任状态机。** 角色产出落库带状态（draft → 人确认 → confirmed），下游角色只消费 confirmed。这是财务现实「单据流转 + 签字盖章」的数字翻版（先例：个税累计预扣接力只用 confirmed 期间数据）。口径类软上下文（公司画像、工作约定）通过注入所有角色的上下文共享，不走消息。
2. **工序协调由主 Agent 编排，人是节拍器。** 跨域活由主 Agent 拆解、派发（spawn_subagent）、汇总。角色遇前置不满足时按 A 段纪律返回「缺什么」，缺口上浮为关注项，由用户决定推进节奏。

**禁止角色间直接通信（A2A），禁止常驻角色进程。** 三条理由：

- 审计黑洞：AI 间私下协商的结果无法回答「这个数是谁算的、基于什么」（红线 3）；
- 错误传播：草稿错误跨域扩散且无人把关——数据层协调强制跨域共享必须过确认门，错误被拦在域内；
- 不解决真问题：域间需要共享的是数（过状态机）和口径（注入上下文），不是聊天。

**工序知识的居所**（按优先序）：主 Agent 领域提示 → 作业技能内容（如 filing-precheck 写明前置核对项）→ 未来的巡检清单。若将来需要多步自动工序，扩展形态是**声明式工序模板（编排即数据，可审查、可版本化）**，仍由主 Agent 执行——不引入网状 A2A，不引入 LangGraph 类框架（与 spec-role-registry 的后置决定同一条线）。

## 后果

- 每一步跨域依赖的处理都可审计、可叫停；
- 长链全自动工序受限：前置不满足时停下等人——对财务场景这是特性（结账推进节奏本就该财务经理控制），接受；
- 角色保持无状态（persistSession: false），工程简单，「角色知道什么」永远可重建。
